# Translation Repository SOP

本仓库用于处理多语言文档翻译任务，当前主要面向 Word 文档。后续所有 agent 和协作者都必须按本 SOP 工作，优先保护源文件、保留原始版式，并通过双轮审校后再交付。

> 必须在 `.worktrees` 目录下创建子分支和工作树进行翻译工作，绝不能在 `main` 分支上操作

## 默认目标

- 源文件格式以 `.docx` 为主。
- 如收到 `.doc`，先转换为 `.docx` 后再进入翻译流程；原始 `.doc` 仍需归档。
- 翻译方向不限，可以是任意源语言到任意目标语言。
- 每个项目开始前必须明确源语言、目标语言和地区变体，例如 `en-US -> zh-CN`、`zh-CN -> en-US`、`ja-JP -> zh-CN`。
- 最终交付物为目标语言 `.docx`。
- 默认质量流程为双轮审校：一轮语言润色，一轮术语、格式和版式 QA。
- 禁止直接覆盖或修改 `source/` 下的原始文件。
- 每个翻译任务必须新建独立 git 分支和独立 worktree，并且只能在 `.worktrees/` 下的对应 worktree 中开展翻译工作。
- 任何读取、创建、编辑或审校 Word 文档的任务，都必须先使用 `.agents/skills/doc/SKILL.md` 中的 `doc` 技能要求。

## 目录约定

```text
source/              # 原始 Word，禁止直接修改（如果用户提供其他路径的文件，你需要拷贝到这里）
work/                # 中间工作文件，如抽取文本、初译稿、审校稿
glossary/            # 术语表、风格指南、客户偏好
output/doc/          # 最终目标语言 Word
output/review/       # 渲染截图、QA 记录、交付说明
tmp/docs/            # 临时转换文件和渲染文件，可清理
```

翻译任务的 worktree 约定：

```text
.worktrees/{项目slug}-{源语言}-{目标语言}/
```

分支命名约定：

```text
translation/{项目slug}-{源语言}-{目标语言}
```

`项目slug` 使用简短 ASCII 名称，空格用 `-`，避免使用特殊符号。若同一项目已有未完成的分支和 worktree，应继续使用原有 worktree，不重复创建。

推荐文件命名：

```text
source/{项目名}/{文档名}.docx
work/{项目名}/{文档名}.{源语言}-{目标语言}.draft.md
work/{项目名}/{文档名}.{源语言}-{目标语言}.reviewed.md
output/doc/{项目名}/{文档名}.{目标语言}.docx
output/review/{项目名}/{文档名}.qa.md
```

语言代码优先使用 BCP 47 风格，例如 `en-US`、`zh-CN`、`zh-TW`、`ja-JP`、`ko-KR`、`fr-FR`。

## 工作流程

1. 立项与归档
   - 明确项目的源语言、目标语言、地区变体、受众和交付要求。
   - 在开始翻译前创建独立分支：`translation/{项目slug}-{源语言}-{目标语言}`。
   - 在 `.worktrees/{项目slug}-{源语言}-{目标语言}/` 创建并进入对应 git worktree。
   - 后续源文件归档、文本抽取、翻译、审校、回写和验收都必须在该 worktree 内完成。
   - 主工作区不得直接进行具体翻译任务，只用于维护仓库级说明、模板和协调信息。
   - 将源语言 Word 原件放入 worktree 内的 `source/{项目名}/`。
   - 不修改 worktree 内 `source/` 下原件。
   - 为每个项目在 worktree 内建立对应的 `work/{项目名}/`、`output/doc/{项目名}/` 和 `output/review/{项目名}/`。
   - `.doc` 文件先用 LibreOffice 转为 `.docx`，转换后的文件进入翻译流程。

2. 预检查
   - 检查文档是否包含目录、页眉页脚、脚注、表格、图片、批注、修订痕迹、文本框或嵌入对象。
   - 尽量将原文档渲染为 PDF/PNG，作为版式对照基准。
   - 在 QA 记录中标出高风险元素，例如复杂表格、SmartArt、扫描图、公式和文本框。

3. 术语与风格准备
   - 在 `glossary/terms.csv` 维护术语，建议字段为 `source,target,source_lang,target_lang,note,status`。
   - 在 `glossary/style-guide.md` 维护翻译风格、固定译法、客户偏好、地区变体和禁用表达。
   - 默认使用目标语言的自然、专业、正式表达；具体语域以项目要求为准。
   - 产品名、代码、API 名称、URL、变量名、占位符和引用编号不得擅自翻译或改写。
   - 不确定术语标记为 `TODO(term)`，不得静默猜测。

4. 文本抽取与初译
   - 优先按段落、标题层级、表格单元格、脚注等结构抽取文本。
   - 初译稿存入 `work/{项目名}/{文档名}.{源语言}-{目标语言}.draft.md` 或同等结构化工作稿。
   - 初译时必须保留占位符、变量、URL、代码、图表编号、脚注编号和交叉引用。
   - 不改变事实、数字、单位、条件、责任主体或逻辑关系。

5. 一轮人工润色
   - 重点检查准确性、语序、专业表达和上下文一致性。
   - 可以按目标语言表达习惯调整句式，但不得改变源文含义。
   - 对无法确认的内容保留 `TODO(check)` 并写入 QA 记录。

6. 二轮审校
   - 检查术语一致性、专名一致性、数字单位、标点、全半角、链接、交叉引用和编号。
   - 清除所有 `TODO(...)`、机器翻译痕迹、占位说明和不自然表达。
   - 审校结果写入 `output/review/{文档名}.qa.md`。

7. Word 回写
   - 将目标语言译文回写为 `.docx`，尽量复用原文档样式、标题层级、表格结构、页眉页脚和分页逻辑。
   - 图片、图注、表注、脚注、目录、编号列表和引用关系必须保留。
   - 最终文件输出到 `output/doc/{项目名}/{文档名}.{目标语言}.docx`。

8. 版式验收
   - 将最终 `.docx` 渲染为 PDF/PNG。
   - 逐页检查文字是否重叠、截断、乱码，表格是否破版，图片和图注是否对齐，页眉页脚是否正常。
   - 与原文档对照章节顺序、图表数量、页码、标题层级和关键格式。
   - 验收结论写入对应 QA 文件。

9. 交付归档
   - 交付 `output/doc/` 下的目标语言 Word。
   - 保留 QA 文件、术语表和必要的渲染截图。
   - `tmp/docs/` 只保存临时文件，最终确认后可清理。

## 质量标准

- 源文件不可被覆盖。
- 译文必须完整，不得漏译标题、表格、脚注、页眉页脚、图注、批注中的正文信息。
- 术语、产品名、组织名和固定表达必须一致。
- 数字、单位、日期、版本号、金额、代码、命令、路径和链接必须逐项核对。
- 不得保留 `TODO(...)`、占位符说明、未处理审校批注或机器翻译提示语。
- 最终 Word 必须可打开、可编辑，并通过逐页渲染检查。

## 工具建议

处理 `.docx` 时必须遵循 `.agents/skills/doc/SKILL.md`：

- 优先使用 `python-docx` 读取、抽取、创建和回写 Word 文档。
- 每次有实质性改动后，应重新渲染并检查页面。
- 如果无法完成视觉渲染检查，必须在交付说明或 QA 记录中明确标出版式风险。
- 中间文件优先放入 `tmp/docs/`，最终 Word 产物放入 `output/doc/`。

优先使用以下工具处理 Word 和版式检查：

```bash
uv pip install python-docx pdf2image
brew install libreoffice poppler
```

- `python-docx`：读取、抽取和回写 `.docx`。
- LibreOffice `soffice`：将 `.doc` 转 `.docx`，或将 `.docx` 渲染为 PDF。
- Poppler `pdftoppm`：将 PDF 转 PNG，用于逐页视觉验收。

如果环境缺少上述工具，应先说明缺失项和影响；在无法渲染检查时，必须明确标出版式风险。

## 验收清单

每个交付文档完成前，至少检查：

- 原文和译文章节结构一致。
- 所有正文、表格、脚注、页眉页脚、图注和列表均已处理。
- 术语表中的译法已全文统一。
- 数字、单位、日期、版本号、金额、链接和代码无误。
- 全文无 `TODO(...)`、占位内容和机器翻译提示语。
- 最终 `.docx` 已渲染并逐页检查，无重叠、截断、乱码或明显破版。
